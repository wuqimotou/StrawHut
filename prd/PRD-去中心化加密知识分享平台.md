## 1. 产品概述

### 1.1 产品定位

StrawHut 是一个**完全运行在本地的知识卡片加密工具**，帮助用户将知识内容加密为 `.straw` 格式的知识卡片文件，并通过密钥控制知识的访问权限。软件不依赖任何中心化服务器，所有数据处理均在用户本地设备完成。

### 1.2 产品愿景

> 让知识在人与人之间安全、自由地传递，创作者掌控内容，读者获得价值。

### 1.3 核心价值主张

| 价值维度       | 描述                                   |
| ---------- | ------------------------------------ |
| **完全本地运行** | 不依赖任何服务器，数据永不离开用户设备（除非用户主动分享）        |
| **端到端加密**  | 知识内容在本地即被加密，只有持有密钥的用户才能解密阅读          |
| **知识卡片化**  | 结构化的 `.straw` 文件格式，便于知识的独立传播与管理      |
| **传播自由**   | 知识卡片的传播与软件解耦，用户可通过任意渠道分享 `.straw` 文件 |
| **隐私保护**   | 支持纯匿名发布模式，完全保护创作者身份                  |

### 1.4 业务目标

- [ ] **短期目标**：完成 Windows 桌面端核心功能开发，实现知识卡片的加密发布与解密读取
- [ ] **中期目标**：扩展至 Android 和 iOS 移动端，完善富文本编辑器和用户体验
- [ ] **长期目标**：评估并可选接入 P2P 传播机制，形成知识卡片生态

### 1.5 产品边界

| 属于 StrawHut 职责 | 不属于 StrawHut 职责            |
| -------------- | -------------------------- |
| 知识内容的编辑与加密     | `.straw` 文件在网络上的传播（用户自行处理） |
| 知识卡片的解密与阅读     | 用户身份验证（无账号系统）              |
| `.straw` 文件的生成与读取 | 知识卡片数据的存储（软件不保存任何卡片/密钥记录） |
| 富文本知识卡片渲染      | 本地存储管理（草稿/密钥/卡片库）          |
| 匿名发布支持        | 在线支付/交易功能（当前版本）            |

> **核心原则**：StrawHut 与知识卡片数据完全解耦，只负责发布（加密生成 .straw 文件）和读取（解密 .straw 文件），不存储任何知识卡片数据、密钥记录或其他持久化信息。

## 2. 目标用户群体

### 2.1 主要用户画像

| 用户角色    | 描述                           | 核心诉求                |
| ------- | ---------------------------- | ------------------- |
| **创作者** | 希望保护知识内容安全、控制传播范围的内容生产者      | 内容加密、匿名发布、便捷导出 .straw 文件 |
| **读者**  | 收到 `.straw` 文件并希望阅读其中知识内容的用户 | 便捷解密、舒适的阅读体验、跨平台支持  |

> **说明**：创作者和读者并非互斥身份，同一用户可以在不同场景下扮演不同角色。

### 2.2 典型使用场景

- **场景一**：技术专家撰写了一份安全研究笔记，使用 StrawHut 编辑内容，系统自动生成密钥并可选择导出 .key 文件，一键生成 `.straw` 文件，通过邮件/即时通讯工具发送给特定受众
- **场景二**：读者收到一份 `.straw` 文件，用 StrawHut 打开并通过手动输入密钥或上传 .key 文件进行解密，解密后阅读完整知识内容
- **场景三**：创作者发布匿名知识卡片，所有作者信息字段均使用匿名标识，知识本身通过传播渠道自由流通
- **场景四**：团队内部使用 StrawHut 加密分享敏感知识文档，无需依赖任何企业内部服务器

### 2.3 用户痛点分析

现有知识分享方式存在的问题：

- 中心化平台可以查看、修改或删除用户内容
- 文件分享缺乏加密保护，内容容易被未授权人员获取
- 知识内容格式不统一，不利于长期保存和传播
- 创作者无法灵活控制内容的访问权限
- 跨平台知识阅读体验不一致

## 3. .straw 知识卡片文件格式规范

### 3.1 文件概述

`.straw` 是 StrawHut 定义的结构化知识卡片文件格式，用于封装加密后的知识内容及其元数据。每个 `.straw` 文件都是一张独立的知识卡片，可被 StrawHut 软件读取和解密。

### 3.2 文件结构

`.straw` 文件采用 JSON 格式封装（外层），内部包含加密内容和元数据字段：

```json
{
  "format_version": "1.0.0",
  "meta": {
    "publisher_alias": "创作者代号或匿名标识",
    "publish_date": "发布日期（ISO 8601 格式）",
    "title": "知识卡片标题",
    "tags": ["标签1", "标签2"],
    "description": "知识卡片简要描述（公开可见，用于预览）",
    "is_anonymous": true,
    "custom_annotations": {
      "自定义字段名": "自定义字段值"
    }
  },
  "content": {
    "encrypted_data": "加密后的知识内容（Base64 编码）",
    "encryption_algorithm": "加密算法标识",
    "iv": "初始化向量（Base64 编码）"
  },
  "integrity": {
    "hash": "文件完整性校验哈希值",
    "hash_algorithm": "哈希算法标识"
  }
}
```

### 3.3 字段详细说明

| 字段路径                           | 类型      | 必填 | 说明                                         |
| ------------------------------ | ------- | -- | ------------------------------------------ |
| `format_version`               | string  | 是  | 文件格式版本号，用于未来兼容性处理                          |
| `meta.publisher_alias`         | string  | 是  | 发布者代号，纯匿名模式下为匿名标识（如 `Anonymous_XXXX`）      |
| `meta.publish_date`            | string  | 是  | 发布日期，ISO 8601 格式（如 `2026-05-01T12:00:00Z`） |
| `meta.title`                   | string  | 是  | 知识卡片标题                                     |
| `meta.tags`                    | array   | 否  | 标签列表，用于知识分类                                |
| `meta.description`             | string  | 否  | 简要描述，未解密时也可展示                              |
| `meta.is_anonymous`            | boolean | 是  | 是否为纯匿名模式                                   |
| `meta.custom_annotations`      | object  | 否  | 其他自定义标注信息                                  |
| `content.encrypted_data`       | string  | 是  | 加密后的知识内容，Base64 编码                         |
| `content.encryption_algorithm` | string  | 是  | 使用的加密算法标识（如 `AES-256-GCM`）                 |
| `content.iv`                   | string  | 是  | 初始化向量，Base64 编码                            |
| `integrity.hash`               | string  | 是  | 文件完整性校验哈希                                  |
| `integrity.hash_algorithm`     | string  | 是  | 哈希算法标识（如 `SHA-256`）                        |

### 3.4 纯匿名模式说明

当 `meta.is_anonymous` 设置为 `true` 时：

- `publisher_alias` 字段使用系统生成的匿名标识（如 `Anonymous_XXXX`）
- `custom_annotations` 中任何可能暴露发布者身份的信息均应替换为匿名值
- 不包含任何可追溯至真实身份的元数据（如 IP 地址、设备指纹等）

### 3.5 知识内容格式

解密后的知识内容为**富文本格式**，支持：

- 文本样式：加粗、斜体、下划线、删除线、颜色
- 标题层级：H1 \~ H6
- 列表：有序列表、无序列表
- 代码块：代码高亮
- 引用块
- 图片（嵌入或链接）
- 表格
- 分隔线

> 具体富文本格式的底层存储方案待确认（见 [待明确事项](#11-待明确事项)）。

## 4. 核心功能模块

### 4.1 知识内容编辑模块

| 功能项    | 优先级 | 描述                 |
| ------ | --- | ------------------ |
| 富文本编辑器 | P0  | 支持所见即所得的富文本内容创作    |
| 标题与段落  | P0  | 支持多级标题和段落排版        |
| 文本样式   | P0  | 加粗、斜体、下划线、删除线、字体颜色 |
| 列表支持   | P0  | 有序列表、无序列表          |
| 代码块    | P1  | 支持代码高亮显示           |
| 引用块    | P1  | 引用内容区块             |
| 图片插入   | P1  | 本地图片或外链图片插入        |
| 表格编辑   | P2  | 支持插入和编辑表格          |
| 预览模式   | P0  | 实时预览最终渲染效果         |
| 内存草稿暂存 | P1  | 编辑过程中内容临时保存在内存中，关闭软件后不保留 |

### 4.2 知识加密发布模块

| 功能项       | 优先级 | 描述                             |
| --------- | --- | ------------------------------ |
| 卡片信息填写    | P0  | 填写标题、发布者代号、描述、标签等元数据           |
| 匿名模式切换    | P0  | 一键切换至纯匿名发布模式                   |
| 随机密钥生成    | P0  | 内容编辑完成后，系统自动生成高强度随机密钥          |
| .key 文件导出 | P0  | 用户可选择是否将密钥导出为独立的 .key 文件       |
| 一键加密导出    | P0  | 使用生成的密钥对内容进行加密，打包为 `.straw` 文件 |
| 文件保存路径选择  | P0  | 用户选择 `.straw` 文件的本地保存位置        |
| 加密算法配置    | P2  | 高级用户可配置加密参数（默认使用推荐配置）          |

#### 加密发布流程

```
1. 用户在富文本编辑器中完成知识内容创作
2. 点击「发布/加密」按钮
3. 填写卡片元信息（标题、代号、描述、标签等）
4. 选择是否开启匿名模式
5. 系统自动生成随机密钥
6. 用户选择是否导出 .key 文件（包含密钥信息，可独立用于解密）
7. 系统使用密钥对内容进行加密
8. 生成文件完整性校验哈希
9. 用户选择保存路径，导出 .straw 文件
10. 完成发布
```

### 4.3 知识解密读取模块

| 功能项          | 优先级 | 描述                         |
| ------------ | --- | -------------------------- |
| 打开 .straw 文件 | P0  | 支持通过文件选择器或拖拽方式导入 .straw 文件 |
| 卡片信息预览       | P0  | 展示未加密的元数据（标题、描述、发布者代号等）    |
| 密钥输入框解密      | P0  | 通过输入框手动输入密钥字符串进行解密         |
| .key 文件上传解密  | P0  | 上传 .key 文件自动读取密钥信息进行解密     |
| 内容解密         | P0  | 使用密钥直接解密知识内容（无需密码派生）       |
| 完整性校验        | P0  | 解密后校验文件完整性，防止篡改            |
| 富文本渲染        | P0  | 将解密后的内容渲染为富文本展示            |
| 解密失败提示       | P0  | 密钥错误或文件损坏时给出明确提示           |

#### 解密读取流程

```
1. 用户通过文件选择器或拖拽方式打开 .straw 文件
2. 软件解析文件，展示卡片元信息预览
3. 用户选择解密方式：
   - 方式一：在输入框中手动输入密钥字符串
   - 方式二：上传 .key 文件，自动读取密钥信息
4. 系统使用密钥直接对加密内容进行解密
5. 执行完整性校验
   - 校验通过：渲染富文本内容并展示
   - 校验失败：提示密钥错误或文件已损坏
6. 用户阅读知识内容
```

## 5. 技术架构

### 5.1 架构原则

- **纯本地运行**：所有功能在用户设备上执行，不依赖任何服务器
- **跨平台**：支持 Windows、Android、iOS 三端
- **数据安全**：用户数据始终保存在本地，不上传

### 5.2 技术栈选型建议

```
┌─────────────────────────────────────────────────────────┐
│                   应用层 (跨平台 UI)                      │
│  富文本编辑器  |  卡片展示组件  |  密钥输入组件            │
├─────────────────────────────────────────────────────────┤
│                   核心服务层                              │
│  加密/解密服务  |  文件I/O服务  |  内存草稿管理服务        │
├─────────────────────────────────────────────────────────┤
│                   数据层                                  │
│  文件系统（.straw 文件读写）                              │
└─────────────────────────────────────────────────────────┘
```

> **说明**： StrawHut 不依赖任何本地数据库或持久化存储，所有数据仅在运行时存在于内存中，软件关闭后完全清空。

### 5.3 跨平台方案对比

| 方案               | 优势                   | 劣势          | 推荐度   |
| ---------------- | -------------------- | ----------- | ----- |
| **Flutter**      | 三端原生性能、单一代码库、UI 一致性高 | Dart 生态相对较小 | ⭐⭐⭐⭐⭐ |
| **Tauri + Web**  | 桌面端体积小、前端生态丰富        | 移动端支持尚在早期   | ⭐⭐⭐   |
| **React Native** | 移动端成熟、生态丰富           | 桌面端支持较弱     | ⭐⭐⭐   |
| **原生开发**         | 各平台最佳体验              | 三套代码、维护成本高  | ⭐⭐    |

> **推荐方案**：Flutter，可一次性覆盖 Windows、Android、iOS 三端，且性能表现优秀。

### 5.4 加密算法选型

| 用途    | 推荐算法            | 说明              |
| ----- | --------------- | --------------- |
| 内容加密  | **AES-256-GCM** | 对称加密，提供加密和完整性校验 |
| 密钥生成  | **CSPRNG**      | 系统自动生成高强度随机密钥   |
| 完整性校验 | **SHA-256**     | 文件哈希校验，防止篡改     |
| 随机数生成 | **CSPRNG**      | 生成 IV 等加密参数     |

### 5.5 富文本编辑器选型

| 方案                          | 跨平台支持 | 与 Flutter 兼容性 | 推荐度      |
| --------------------------- | ----- | ------------- | -------- |
| **Quill**（通过 WebView）       | 三端    | 需桥接           | ⭐⭐⭐⭐     |
| **Flutter Quill**           | 三端    | 原生            | ⭐⭐⭐⭐⭐    |
| **ProseMirror**（通过 WebView） | 三端    | 需桥接           | ⭐⭐⭐      |
| **自定义实现**                   | 三端    | 原生            | ⭐⭐（工作量大） |

> **推荐方案**：Flutter Quill，基于 Quill 的 Flutter 原生实现，功能丰富且跨平台兼容性好。

## 6. 安全性和隐私保护

### 6.1 安全要求

| 安全维度       | 要求                  | 实现方式              |
| ---------- | ------------------- | ----------------- |
| **内容加密**   | 知识内容使用强加密算法保护       | AES-256-GCM，本地加密  |
| **密钥安全**   | 系统生成高强度随机密钥，不依赖用户密码 | CSPRNG 生成，使用时在内存中处理，关闭后清除 |
| **完整性保护**  | 防止文件被篡改             | SHA-256 完整性校验     |
| **内存安全**   | 敏感数据使用后立即清理        | 密钥、明文内容使用后立即清除内存   |
| **无数据上传**  | 所有数据仅在本地处理          | 架构设计保证，无网络请求模块    |
| **无持久化存储** | 软件不保存任何卡片、密钥或草稿记录  | 关闭软件后内存完全清空       |

### 6.2 隐私保护要求

- [ ] **纯匿名发布**：创作者可选择匿名模式，所有身份信息使用匿名标识替代
- [ ] **无用户追踪**：软件不收集任何用户行为数据
- [ ] **无身份绑定**：无需注册账号，无需关联真实身份
- [ ] **本地化运行**：所有操作在本地完成，数据不离开用户设备
- [ ] **无痕阅读**：不强制记录用户阅读历史（可选记录用于管理）

### 6.3 威胁模型

| 威胁类型         | 描述                       | 防护措施                                |
| ------------ | ------------------------ | ----------------------------------- |
| 密钥泄露         | 用户不慎将密钥或 .key 文件分享给未授权人员 | 提醒用户妥善保管密钥，.key 文件不自动附带在 .straw 文件中 |
| .key 文件泄露    | .key 文件被他人获取后直接用于解密      | 用户需注意 .key 文件的存储和传输安全               |
| 文件篡改         | .straw 文件被恶意修改           | SHA-256 完整性校验，解密时验证                 |
| 内存泄露         | 软件运行期间内存被恶意程序访问          | 敏感数据使用后立即清除内存，降低攻击窗口                |
| 侧信道攻击        | 通过内存分析获取密钥               | 使用后及时清除内存中的密钥数据                     |
| 钓鱼 .straw 文件 | 恶意构造的 .straw 文件          | 文件格式校验，异常结构拦截                       |

## 7. 用户角色和权限设计

### 7.1 角色定义

| 角色      | 描述                                 | 权限范围                          |
| ------- | ---------------------------------- | ----------------------------- |
| **创作者** | 使用 StrawHut 编辑知识内容并加密发布为 .straw 文件 | 内容编辑、卡片元信息填写、密钥生成、加密导出、匿名模式切换 |
| **读者**  | 使用 StrawHut 打开并解密 .straw 文件以阅读知识内容 | 文件导入、密钥输入、内容解密、富文本阅读          |

> **说明**：创作者和读者是功能角色，同一用户可同时使用两种角色功能。

### 7.2 权限矩阵

| 操作             | 创作者 | 读者 |
| -------------- | --- | -- |
| 编辑知识内容         | ✅   | ❌  |
| 加密发布 .straw 文件 | ✅   | ❌  |

## 9. 传播机制设计

### 9.1 传播与软件解耦

StrawHut 软件本身**不负责** `.straw` 文件的传播。知识卡片的传播完全由用户自主决定，可通过任意渠道进行：

| 传播渠道 | 示例                                 |
| ---- | ---------------------------------- |
| 即时通讯 | 微信、Telegram、Signal、WhatsApp 等      |
| 电子邮件 | 附件发送                               |
| 云存储  | Google Drive、Dropbox、OneDrive、坚果云等 |
| 社交媒体 | 网盘链接分享                             |
| 物理介质 | U 盘、移动硬盘                           |
| 其他   | 任何用户认为合适的方式                        |

### 9.2 设计原则

1. **软件边界清晰**：StrawHut 仅负责知识卡片的创建（加密）和消费（解密），传播是用户行为
2. **文件格式开放**：`.straw` 是标准化的文件格式，不依赖特定软件版本即可被未来兼容的软件读取
3. **零网络依赖**：软件无需联网即可完成所有核心操作
4. **传播方式自由**：不限制、不绑定任何传播渠道

### 9.3 传播流程图

```
创作者                            读者
  │                                │
  ├── 编辑知识内容                  │
  ├── 加密生成 .straw 文件          │
  │                                │
  ═══ [ .straw 文件通过任意渠道传播 ] ═══
         （微信 / 邮件 / 云盘 / ...）
  │                                │
  │                   接收到 .straw 文件
  │                   用 StrawHut 打开
  │                   输入密钥解密
  │                   阅读知识内容
```

## 10. 未来规划：P2P 传播升级（可选）

> **说明**：以下内容仅为远期规划参考，**不纳入当前版本开发范围**。

### 10.1 升级目标

在保留现有本地工具核心功能的前提下，可选接入 P2P 传播机制，使 `.straw` 知识卡片能够在用户之间自动发现和传播，形成去中心化的知识网络。

### 10.2 可能的 P2P 技术方案

| 方案                          | 说明                         | 优势       |
| --------------------------- | -------------------------- | -------- |
| **IPFS**                    | 将加密后的 .straw 文件存储到 IPFS 网络 | 成熟、抗审查   |
| **BitTorrent / WebTorrent** | 通过 BT 协议分发知识卡片             | 高并发、去中心化 |
| **自定义 P2P 网络**              | 基于 libp2p 构建 StrawHut 专属网络 | 定制化强     |
| **混合方案**                    | 本地工具 + 可选 P2P 插件           | 灵活、渐进式升级 |

### 10.3 升级条件

- 核心本地工具功能稳定成熟
- 社区对 P2P 传播有明确需求
- 技术可行性和合规性得到验证
- 不影响现有 `.straw` 文件格式的兼容性
