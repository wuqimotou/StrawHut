## 1 选型原则

本项目技术选型遵循以下核心原则：

| 原则 | 说明 |
|------|------|
| **跨平台优先** | 一套代码覆盖 Windows、Android、iOS 三端，降低开发和维护成本 |
| **原生性能** | 关键组件（编辑器、加解密）必须使用原生实现，避免 WebView 性能损耗 |
| **安全合规** | 加密算法采用业界标准，经过充分验证和广泛使用 |
| **生态完善** | 优先选择社区活跃、文档完善、长期维护的技术方案 |
| **轻量简洁** | 避免引入不必要的复杂依赖，保持项目轻量化 |
| **可扩展性** | 技术栈需支持未来 P2P 升级、多语言国际化等扩展需求 |
| **开发效率** | 考虑团队学习曲线，降低上手门槛 |

## 3 跨平台框架选型

### 3.1 候选方案对比

| 方案 | 优势 | 劣势 | 适用场景 | 推荐度 |
|------|------|------|----------|--------|
| **Flutter** | 三端原生性能、单一代码库、UI 一致性高、Hot Reload 开发效率高 | Dart 生态相对较小、首包体积较大 | 跨平台应用、需要原生渲染性能 | ⭐⭐⭐⭐⭐ |
| Tauri v2 | 桌面端体积小巧、Web 技术栈丰富 | 移动端支持尚不成熟、社区较小 | 以桌面端为主的应用 | ⭐⭐⭐ |
| React Native | 移动端生态成熟、Web 技术栈 | 桌面端支持较弱、原生模块桥接复杂 | 以移动端为主的应用 | ⭐⭐⭐ |
| 原生开发 (Swift/Kotlin/C++) | 各平台最佳性能与体验 | 三套代码库、维护成本极高、开发周期长 | 对平台体验要求极高的应用 | ⭐⭐ |

### 3.2 推荐方案：Flutter

**选型理由：**

1. **三端覆盖能力**：Flutter 是目前唯一能同时以较高成熟度支持 Windows、Android、iOS 的跨平台框架
2. **原生渲染性能**：通过 Skia/Impeller 引擎直接渲染，不依赖 WebView，编辑器响应延迟可满足 < 50ms 要求
3. **统一开发体验**：单一代码库、单一语言（Dart），大幅降低三端适配成本
4. **热重载（Hot Reload）**：提升开发效率，UI 调整实时可见
5. **丰富的插件生态**：加密、文件操作、状态管理等核心需求均有成熟插件支持
6. **Google 官方维护**：长期支持有保障，版本迭代稳定

**版本建议：** Flutter 3.x 稳定版本（推荐 3.24+）

**平台兼容性：**

| 平台 | 最低支持版本 | 目标版本 | 注意事项 |
|------|-------------|---------|----------|
| Windows | Windows 10 (Build 19041+) | Windows 11 | 需安装 Visual Studio Build Tools |
| Android | Android 8.0 (API 26) | Android 14+ | 需配置 Gradle、Android SDK |
| iOS | iOS 13.0 | iOS 17+ | 需 macOS 环境和 Xcode |

**风险点：**
- Dart 生态相比 JavaScript/TypeScript 较小，部分场景可能需要自行实现
- iOS 端需要 macOS 开发环境
- Windows 端桌面应用打包需注意签名和分发

## 4 富文本编辑器选型

### 4.1 候选方案对比

| 方案 | 跨平台支持 | Flutter 兼容性 | 性能表现 | 功能丰富度 | 维护状态 | 推荐度 |
|------|-----------|---------------|----------|-----------|---------|--------|
| **flutter_quill** | 三端原生 | 原生 Flutter Widget | 优秀（无 WebView 开销） | 丰富（支持 Quill 所有功能） | 活跃 | ⭐⭐⭐⭐⭐ |
| Quill (WebView 桥接) | 三端 | 需 WebView 桥接 | 一般（WebView 性能损耗） | 丰富 | 活跃 | ⭐⭐⭐ |
| ProseMirror (WebView 桥接) | 三端 | 需 WebView 桥接 | 一般 | 丰富 | 活跃 | ⭐⭐⭐ |
| flutter_z_editor | 三端 | 原生 | 良好 | 基础功能 | 一般 | ⭐⭐ |
| 自定义实现 | 三端 | 原生 | 可控 | 取决于开发投入 | - | ⭐⭐（开发成本极高） |

### 4.2 推荐方案：flutter_quill

**选型理由：**

1. **Quill Delta 格式**：原生支持 Delta 格式，可直接序列化为 JSON，完美匹配 `.straw` 文件中富文本内容的存储需求
2. **Flutter 原生渲染**：无 WebView 桥接开销，响应延迟可控制在 50ms 以内
3. **功能覆盖全面**：
   - 支持文本样式：加粗、斜体、下划线、删除线、字体颜色
   - 支持标题层级：H1 ~ H6
   - 支持列表：有序列表、无序列表
   - 支持代码块、引用块、分隔线
   - 支持图片插入（Base64 嵌入或 URL 引用）
   - 支持表格（通过扩展插件）
4. **自定义工具栏**：可根据 StrawHut 需求定制工具栏按钮
5. **活跃维护**：GitHub 社区活跃，定期更新，Bug 修复及时

**版本建议：** flutter_quill 最新稳定版本（推荐 10.x+）

**Delta 格式存储建议：**

StrawHut 中富文本内容采用 **Delta JSON 格式** 作为底层存储方案，原因如下：

- Delta 是 Quill 编辑器的原生数据格式，无需转换即可存储和恢复
- JSON 格式具有良好的可读性和可扩展性
- 支持富文本操作语义化，方便未来增加协作编辑等功能
- 加密前将 Delta JSON 序列化，加密后存入 `content.encrypted_data` 字段

**风险点：**
- 表格编辑功能需要通过额外扩展实现，可能需要定制开发
- 大文档（包含大量图片 Base64）场景下需要关注性能表现

## 5 加密算法选型

### 5.1 算法选型矩阵

| 用途 | 推荐算法 | 密钥长度 | 说明 | 推荐度 |
|------|---------|---------|------|--------|
| **内容加密** | AES-256-GCM | 256 bit | 对称加密，提供加密 + 认证（完整性保护） | ⭐⭐⭐⭐⭐ |
| **完整性校验** | SHA-256 | 256 bit | 文件级哈希校验，防篡改验证 | ⭐⭐⭐⭐⭐ |
| **密钥生成** | CSPRNG | 256 bit (32 字节) | 密码学安全伪随机数生成器 | ⭐⭐⭐⭐⭐ |
| **随机数（IV/Nonce）** | CSPRNG | 96 bit (12 字节) | GCM 模式标准 Nonce 长度 | ⭐⭐⭐⭐⭐ |

### 5.2 内容加密：AES-256-GCM

**算法特性：**

- **AES (Advanced Encryption Standard)**：NIST 标准，经过广泛验证和审查
- **256 位密钥**：提供极高的安全强度，抗暴力破解
- **GCM (Galois/Counter Mode)**：认证加密模式，同时提供机密性和完整性保护
- **12 字节 Nonce**：GCM 推荐标准长度，避免 Nonce 重用风险

**为什么选择 GCM 而非 CBC：**

| 对比维度 | AES-256-GCM | AES-256-CBC |
|---------|-------------|-------------|
| 完整性保护 | 内置（Auth Tag） | 需要额外 HMAC |
| 并行加密 | 支持 | 不支持 |
| 性能 | 更高（硬件加速） | 较低 |
| 安全性 | 认证加密，防篡改 | 需额外处理填充攻击 |
| 实现复杂度 | 低 | 较高 |

### 5.3 完整性校验：SHA-256

**用途说明：**

- 对整个 `.straw` 文件内容（不含 `integrity` 字段本身）计算 SHA-256 哈希
- 写入 `integrity.hash` 字段
- 解密时重新计算并比对，验证文件是否被篡改

**与 AES-256-GCM 的关系：**

AES-256-GCM 本身提供密文完整性保护（Auth Tag），但文件级 SHA-256 校验是额外的一层防护：
- GCM 的 Auth Tag 保护的是加密数据本身
- SHA-256 保护的是整个 `.straw` 文件结构（包括 `meta` 元数据等未加密字段）
- 两者互补，提供更全面的安全保障

### 5.4 密钥生成：CSPRNG

**实现方式：**

使用 Dart `crypto` 包的 `Random.secure()` 或系统级随机数源：

- 生成 32 字节（256 位）随机密钥
- 使用 Base64 编码（URL-safe 变体）转换为字符串，便于用户复制和输入
- 编码后密钥长度约 43 个字符

**密钥展示格式：**

```
示例密钥：xK9mP2vR7wN4jQ8tL5yH3bF6dA1cE0gU6sZ9oI2eM=
```

**密钥强度：**

- 256 位熵，远超安全阈值
- 暴力破解不可行（2^256 种可能）
- 无需密码派生，直接使用

### 5.5 加密相关 Flutter 依赖

| 用途 | 推荐包 | pub.dev 链接 | 说明 |
|------|--------|-------------|------|
| AES 加密 | `encrypt` | https://pub.dev/packages/encrypt | 封装 PointyCastle，提供简洁的 AES-GCM API |
| 底层密码学 | `pointycastle` | https://pub.dev/packages/pointycastle | Bouncy Castle 的 Dart 实现，AES 底层引擎 |
| SHA-256 哈希 | `crypto` | https://pub.dev/packages/crypto | 官方维护的加密哈希库 |
| 随机数生成 | Dart SDK 内置 | `dart:math` 中的 `Random.secure()` | 系统级 CSPRNG |

**为什么选择 `encrypt` 包而非直接使用 `pointycastle`：**

- `encrypt` 提供了更简洁的高级 API，减少出错风险
- 自动处理 Nonce 生成、认证标签等细节
- 代码可维护性更高

## 6 状态管理方案选型

### 6.1 候选方案对比

| 方案 | 学习曲线 | 样板代码量 | 性能 | 可测试性 | 社区活跃度 | 推荐度 |
|------|---------|-----------|------|---------|-----------|--------|
| **Riverpod** | 中等 | 少 | 优秀 | 优秀 | 活跃 | ⭐⭐⭐⭐⭐ |
| Bloc/Cubit | 中等偏高 | 较多 | 良好 | 优秀 | 活跃 | ⭐⭐⭐⭐ |
| Provider | 低 | 少 | 良好 | 良好 | 活跃 | ⭐⭐⭐ |
| GetX | 低 | 少 | 良好 | 一般 | 活跃 | ⭐⭐⭐ |
| MobX | 中等 | 中等 | 良好 | 良好 | 一般 | ⭐⭐⭐ |

### 6.2 推荐方案：Riverpod 2.x

**选型理由：**

1. **编译时安全**：Riverpod 2.x 引入 Code Generation，编译时检测 Provider 依赖错误，避免运行时异常
2. **依赖注入友好**：天然支持依赖注入，便于单元测试和模块解耦
3. **无 BuildContext 依赖**：可在 UI 树外部访问状态，适合加密/解密等非 UI 场景
4. **状态作用域清晰**：Provider 的作用域机制天然适配 StrawHut 的模块化设计
5. **轻量简洁**：样板代码少，不引入过多概念
6. **官方推荐趋势**：Riverpod 是 Provider 作者 Dan 的继任项目，代表了 Flutter 状态管理的未来方向

**StrawHut 中的使用场景：**

| 状态类型 | Provider 类型 | 说明 |
|---------|--------------|------|
| 编辑器内容状态 | StateNotifierProvider | 富文本 Delta 内容的内存草稿状态 |
| 加密/解密服务 | Provider | 单例注入加密服务实例 |
| 当前打开的卡片 | StateProvider | 当前加载的 `.straw` 文件解析结果 |
| 主题设置 | StateProvider | 应用主题（亮色/暗色） |
| 语言设置 | StateProvider | 国际化语言选择 |

**不选择 Bloc 的理由：**

- Bloc 需要定义大量 Event 和 State 类，样板代码较多
- 对于 StrawHut 这种轻量级本地应用，Bloc 的复杂度偏高
- Riverpod 在保持可测试性的同时，代码更简洁

## 7 文件 I/O 方案选型

### 7.1 文件操作场景分析

| 操作场景 | 说明 | 技术要求 |
|---------|------|---------|
| 导出 .straw 文件 | 保存加密后的知识卡片到用户指定位置 | 文件写入、路径选择 |
| 导入 .straw 文件 | 从用户指定位置读取知识卡片文件 | 文件读取、路径选择 |
| 导出 .key 文件 | 保存密钥文件到用户指定位置 | 文件写入、路径选择 |
| 导入 .key 文件 | 读取密钥文件内容 | 文件读取、路径选择 |
| 拖拽导入 | 支持桌面端拖拽 .straw 文件到应用窗口 | 桌面端特有功能 |

### 7.2 候选方案对比

| 功能 | 推荐包 | 适用平台 | 说明 | 推荐度 |
|------|--------|---------|------|--------|
| 文件选择（导入） | `file_picker` | 全平台 | 统一的文件选择器 API，支持文件类型过滤 | ⭐⭐⭐⭐⭐ |
| 文件保存（导出） | `file_selector` | 全平台 | 支持保存文件对话框，三端一致 API | ⭐⭐⭐⭐⭐ |
| 路径访问 | `path_provider` | 全平台 | 获取系统标准路径（下载目录、临时目录等） | ⭐⭐⭐⭐⭐ |
| 文件系统操作 | Dart `dart:io` | 全平台 | Dart SDK 内置文件读写 API | ⭐⭐⭐⭐⭐ |
| 拖拽支持（桌面） | `desktop_drop` | Windows/macOS/Linux | 桌面端拖拽文件支持 | ⭐⭐⭐⭐ |

### 7.3 推荐方案组合

**文件导入流程：**

```
用户点击"打开文件" 
  → file_selector.openFile() 弹出系统文件选择器
  → 过滤 .straw 文件类型
  → 用户选择文件后返回 File 对象
  → dart:io 的 File.readAsString() 读取文件内容
```

**文件导出流程：**

```
用户点击"保存文件"
  → file_selector.getSaveLocation() 弹出保存对话框
  → 用户选择保存路径
  → dart:io 的 File.writeAsString() 写入 .straw JSON 内容
```

**.key 文件操作：**

```
与 .straw 文件操作相同，仅文件扩展名不同（.key）
```

**拖拽支持（Windows 桌面端）：**

```
使用 desktop_drop 包监听拖入事件
  → 获取拖入的 .straw 文件路径
  → 读取并解析文件内容
```

- 密钥生成的随机性（统计测试）
- 完整性校验的篡改检测能力
- 异常文件格式的容错处理
- 内存安全相关测试（验证敏感数据是否被正确清除）

## 11 风险评估与替代方案

### 11.1 技术风险矩阵

| 风险项 | 影响程度 | 发生概率 | 缓解措施 |
|--------|---------|---------|---------|
| flutter_quill 停止维护 | 高 | 低 | 关注社区动态，必要时 fork 或迁移 |
| encrypt 包发现安全漏洞 | 高 | 低 | 可切换为直接调用 pointycastle |
| Dart 生态不够完善 | 中 | 中 | 自行实现或寻找替代方案 |
| iOS 端审核合规问题 | 中 | 中 | 提前了解 App Store 加密应用合规要求 |
| 大文件加密性能瓶颈 | 中 | 低 | 分块加密、异步处理 |
| 三端 UI 不一致 | 低 | 中 | 使用平台适配组件（Material/Cupertino） |

### 11.2 替代方案速查

| 组件 | 首选方案 | 备选方案 | 切换条件 |
|------|---------|---------|---------|
| 跨平台框架 | Flutter | React Native (移动端优先时) | Flutter 无法满足性能需求 |
| 富文本编辑器 | flutter_quill | 自定义编辑器 | flutter_quill 无法满足定制需求 |
| 状态管理 | Riverpod | Bloc/Cubit | 团队更熟悉 Bloc 模式 |
| 加密库 | encrypt | pointycastle 直接调用 | encrypt 出现兼容性问题 |
| 文件选择 | file_picker | 平台原生文件选择 | file_picker 平台支持不足 |

## 12 选型决策总结

### 12.1 核心技术栈决策

| 决策项 | 选择 | 核心理由 |
|--------|------|---------|
| 跨平台框架 | **Flutter** | 三端覆盖、原生性能、开发效率高 |
| 富文本编辑器 | **flutter_quill** | Delta 格式原生支持、Flutter 原生渲染 |
| 加密算法 | **AES-256-GCM** | 业界标准、认证加密、硬件加速 |
| 完整性校验 | **SHA-256** | 广泛验证、与 GCM Auth Tag 互补 |
| 密钥生成 | **CSPRNG** | 系统级安全随机源 |
| 状态管理 | **Riverpod 2.x** | 编译时安全、轻量简洁 |
| 开发语言 | **Dart 3.x** | Flutter 配套、现代化语言特性 |

### 12.2 与 PRD 对齐验证

| PRD 要求 | 技术选型结果 | 状态 |
|---------|-------------|------|
| 跨平台 Windows/Android/iOS | Flutter | ✅ 满足 |
| 富文本编辑器 | flutter_quill | ✅ 满足 |
| AES-256-GCM 内容加密 | encrypt 包 | ✅ 满足 |
| SHA-256 完整性校验 | crypto 包 | ✅ 满足 |
| CSPRNG 密钥生成 | Dart `Random.secure()` | ✅ 满足 |
| 纯本地运行 | 无网络依赖设计 | ✅ 满足 |
| 无持久化存储 | 内存草稿 + 文件 I/O | ✅ 满足 |
| 编辑器响应 < 50ms | Flutter 原生渲染 | ✅ 预期满足 |
| 加密导出 < 3s | AES-256-GCM 硬件加速 | ✅ 预期满足 |
| 解密渲染 < 2s | 异步解密 + 即时渲染 | ✅ 预期满足 |
| 应用启动 < 2s | Flutter AOT 编译 | ✅ 预期满足 |
| P2P 可扩展性 | 预留接口层 | ✅ 架构支持 |
